# StarVista Project Brief

Date: 2026-06-13

GitHub repository: [MightyKartz/StarVista](https://github.com/MightyKartz/StarVista.git)

## 1. Project Name

**StarVista**

Meaning:

- **Star**: GitHub stars and high-interest open-source projects.
- **Vista**: A broad visual view, panorama, or scenic overview.

Working tagline:

> Turn high-star GitHub repositories into visual project stories.

Chinese positioning:

> 用视觉方式浏览、理解和发现值得关注的开源项目。

## 2. Core Idea

GitHub projects are still mostly introduced through text-heavy README files, stars, topics, and repository metadata. This makes discovery useful but visually dry. StarVista aims to automatically discover high-star or fast-growing GitHub repositories and generate visual, lightweight, shareable introductions for them.

The first version should stay simple:

- No full video generation.
- No complex code understanding.
- No real-time large model dependency.
- No backend server required for user traffic.

Instead, StarVista should generate static visual project cards and project pages from GitHub metadata, README text, repository structure, languages, topics, releases, and daily snapshots.

## 3. Product Positioning

StarVista is not primarily a code analysis tool. It is a visual open-source discovery tool.

Good positioning:

> GitHub Trending + Star History + visual project cards + lightweight project stories.

Avoid positioning:

- "Automatically understand any codebase perfectly."
- "Generate complete architecture documentation."
- "Create AI videos for every repository."

The early product should help users answer:

- What is this project?
- Why is it popular or worth watching?
- What is it built with?
- Is it active?
- What does the repository look like structurally?
- How has it grown over time?

## 4. Target Users

Primary users:

- Developers browsing interesting open-source projects.
- Open-source enthusiasts tracking high-star or fast-growing repositories.
- Builders looking for inspiration from popular projects.

Secondary users:

- Open-source maintainers who want a visual card or profile for their own project.
- Newsletter writers or content creators who summarize trending repositories.
- New contributors trying to quickly understand whether a project is approachable.

## 5. Similar Projects And References

Existing tools are adjacent but not exactly the same:

- [GitDiagram](https://github.com/ahmedkhaleel2004/gitdiagram): Converts a GitHub repository into an interactive architecture diagram. Close to repository visualization, but focused on codebase structure rather than project discovery.
- [GitVizz](https://github.com/adithya-s-k/GitVizz): Visualizes and analyzes GitHub or local repositories with summaries, file structure, and dependency graphs. More code-analysis oriented.
- [githubocto/repo-visualizer](https://github.com/githubocto/repo-visualizer): A GitHub Action that creates an SVG diagram of a repository. Useful reference for low-cost static visualization.
- [Star History](https://www.star-history.com/): Shows GitHub star growth charts. Strong for popularity trends, but does not explain the project.
- [github-stars-video](https://github.com/scastiel/github-stars-video): Generates animated videos for repository star history. Useful reference for later animation ideas.
- [OSSInsight](https://ossinsight.io/): Analyzes GitHub event data at ecosystem scale. Strong data platform, but less focused on visual project storytelling.
- [Trendshift](https://trendshift.io/): Discovers trending open-source projects and momentum. Closer to discovery, but not focused on generated visual introductions.
- [GitHub Pages](https://docs.github.com/en/pages/getting-started-with-github-pages/what-is-github-pages): Static hosting option for the project site.
- [GitHub Actions schedule](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax#onschedule): Can run the data pipeline daily.
- [GitHub REST API rate limits](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api): Important for designing caching and incremental updates.

Opportunity:

Codebase diagrams, star charts, and open-source analytics already exist. The gap is an automatic visual discovery experience that turns changing high-star repositories into easy-to-browse project stories.

## 6. Recommended MVP

The first version should be a static website hosted on GitHub Pages.

MVP features:

1. Daily generated list of high-star or trending repositories.
2. Visual repository cards on the homepage.
3. Project detail page for each repository.
4. Language/topic filters.
5. Lightweight trend indicators from daily snapshots.
6. Static JSON data generated by GitHub Actions.

Do not build these in MVP:

- Real-time repository analysis.
- Full video generation.
- User accounts.
- Comments, likes, or social features.
- Large-scale database.
- Mandatory large model calls.
- Deep dependency analysis across all files.

## 7. Visual Card Content

Each repository card should include:

- Repository name and owner.
- Short description.
- Primary language.
- Stars, forks, open issues.
- Last pushed date.
- Topics.
- License if available.
- Compact language breakdown.
- Activity signal such as active, cooling, fast-rising, stable, or archived.
- Visual accent based on language or category.

Optional later additions:

- Mini star-growth sparkline.
- Maintainer avatar cluster.
- Release freshness badge.
- Project category badge.
- "Why it matters" generated sentence.

## 8. Project Detail Page

Each project detail page should include:

- Hero summary: name, description, stars, language, owner.
- "What it is" section generated from GitHub description and README opening.
- Tech stack section inferred from language statistics and manifest files.
- Repository structure map from the top-level file tree.
- Activity timeline based on releases, pushes, and snapshot history.
- Growth chart once enough daily snapshots exist.
- Links to GitHub repository, releases, issues, and documentation.

The detail page should feel like a visual profile, not a long text article.

## 9. Technical Architecture

Recommended architecture:

```text
GitHub Actions scheduled workflow
        |
        v
Fetch repository list and metadata
        |
        v
Fetch README, languages, topics, releases, file tree
        |
        v
Generate normalized JSON data
        |
        v
Build static frontend
        |
        v
Deploy to GitHub Pages
```

Suggested stack:

- Frontend: Astro or Vite + React.
- Language: TypeScript.
- GitHub API client: Octokit.
- Charts: ECharts, D3, or Observable Plot.
- Animation: CSS transitions first, Framer Motion later if needed.
- Hosting: GitHub Pages for MVP.
- Data storage: JSON files committed or generated during build.

Astro is a strong candidate because the project is content-heavy, static-first, and can still include interactive islands where useful.

## 11. Data Pipeline

Daily workflow:

1. Query GitHub for high-star or recently active repositories.
2. Normalize repository metadata.
3. For each selected repository, fetch:
   - README.
   - language statistics.
   - topics.
   - releases.
   - top-level file tree.
   - owner metadata.
4. Generate visual-friendly JSON.
5. Write a daily snapshot.
6. Build the static site.
7. Deploy to GitHub Pages.

Example data files:

```text
public/data/manifest.json
public/data/repos/vercel-next.js.json
public/data/repos/rust-lang-rust.json
public/data/snapshots/2026-06-13.json
```

## 12. Large Model Strategy

The project should not require real-time LLM calls.

Recommended strategy:

```text
Default mode: no LLM
Optional mode: offline LLM generation during GitHub Actions
Advanced mode: user provides their own key
```

Default non-LLM generation can use:

- Repository description.
- README first paragraph.
- Topics.
- Language statistics.
- Package manifest detection.
- Release notes metadata.
- Simple templates.

Example generated copy:

```text
{repo} is a {primaryLanguage} project focused on {description}.
It has {stars} stars, was last updated on {lastPush}, and uses {topics}.
```

If LLM support is added later:

- Run only during scheduled generation, never on page load.
- Cache generated summaries by repository commit SHA or README hash.
- Regenerate only when the README or repository metadata changes.
- Make the feature optional.

## 13. GitHub Pages Deployment

The project can be hosted directly on GitHub using GitHub Pages.

Expected URL:

```text
https://<username>.github.io/StarVista/
```

Or with a custom domain later.

Important constraints:

- GitHub Pages hosts static files only.
- No backend server runs on GitHub Pages.
- Dynamic content should be generated by GitHub Actions ahead of time.
- README embeds should use SVG, PNG, GIF, or Mermaid-style static output, not complex JavaScript.

Recommended MVP deployment:

```text
GitHub repository = source code and data pipeline
GitHub Actions = daily data update and static build
GitHub Pages = public visual website
```

## 14. API And Rate Limit Considerations

GitHub API usage should be conservative.

Design choices:

- Use authenticated requests in GitHub Actions.
- Use the built-in `GITHUB_TOKEN` first.
- Cache previous repository metadata.
- Avoid fetching deep file trees for every repository every day.
- Update a small number of repositories per run at first.
- Store daily snapshots to avoid rebuilding history from scratch.
- Back off on rate-limit responses.

Initial scale:

- Top 50 to 100 repositories per day.
- Fetch details only for newly discovered or changed repositories.
- Keep snapshots compact.

## 15. Repository Selection Strategy

Possible inputs:

- Most-starred repositories overall.
- Recently pushed repositories above a star threshold.
- GitHub Trending scrape or manual seed list.
- Topic-based lists such as AI, CLI, database, frontend, self-hosted.

Recommended first approach:

```text
Use GitHub API search with stars and recent activity filters.
Limit to a manageable number of repositories.
Group by primary language and topic.
```

Later:

- Add curated collections.
- Add daily rising projects.
- Add weekly "fastest growing" projects.
- Add user-configurable topic feeds.

## 16. Visual Direction

The UI should feel like a visual discovery surface, not an admin dashboard.

Recommended style:

- Dense but readable project cards.
- Strong typography.
- Language color accents.
- Small, useful charts.
- Smooth hover states.
- Subtle animated transitions.
- Clear filters and sorting.

Avoid:

- Marketing landing page as the first screen.
- Decorative visuals that do not explain the project.
- Overly flashy animation.
- One-color theme.
- Text-heavy cards.

The homepage should immediately show actual repositories.

## 17. Roadmap

Phase 1: Static Visual Cards

- Build data fetcher.
- Generate normalized JSON.
- Render homepage cards.
- Deploy to GitHub Pages.

Phase 2: Project Detail Pages

- Add detail pages.
- Add repository structure map.
- Add language and activity charts.
- Add topic filters.

Phase 3: Daily Snapshots

- Save daily snapshots.
- Show growth indicators.
- Add "new today" and "rising" views.

Phase 4: Embeddable Assets

- Generate SVG project cards.
- Add README embed examples.
- Add social sharing image generation.

Phase 5: Optional AI Summaries

- Add optional offline summary generation.
- Cache by README hash.
- Support user-provided API key.

Phase 6: Deeper Visual Stories

- Add richer timeline views.
- Add animated star-growth cards.
- Add category-specific visual templates.
- Explore short generated videos only after the static experience is useful.

## 18. Open Questions

Questions to decide before implementation:

- Should the first feed focus on all-time high-star repositories or newly rising repositories?
- Should the first version support only public GitHub repositories?
- Should GitHub Trending be scraped, or should the project rely only on official API search?
- Should generated data be committed into the repository or only produced as build artifacts?
- Should the site be English-first, Chinese-first, or bilingual?
- Should the product target project discovery first or README/embed assets first?

Recommended defaults:

- Public GitHub repositories only.
- English-first UI with room for Chinese later.
- Official GitHub API first.
- Static JSON generated during Actions.
- Focus on discovery first, embed assets later.

## 19. Success Criteria For MVP

The MVP is successful if:

- The site updates automatically once per day.
- It displays at least 50 real repositories.
- Each repository has a useful visual card.
- Users can filter by language or category.
- Project pages are generated without manual writing.
- No runtime LLM calls are needed.
- The whole project can run from GitHub Actions and GitHub Pages.

## 20. Short Development Summary

StarVista should begin as a low-cost, static, automatically updated visual browser for notable GitHub repositories. The technical bet is simple: generate structured data offline, then make the frontend delightful enough that browsing open-source projects feels more like exploring a gallery than reading a directory.
